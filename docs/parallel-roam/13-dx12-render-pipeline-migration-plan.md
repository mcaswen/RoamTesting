# DX12 渲染与计算管线迁移计划

> 日期：2026-07-15  
> 状态：迁移规划初稿，评审通过后按阶段实施  
> 目标：为 CBT 2024 忠实复现建立与官方实现接近的 DX12/HLSL 实验环境，同时保留现有 CPU ROAM、数据导向 CPU ROAM 和自动基准测试能力

## 1. 决策摘要

项目主开发管线计划从 OpenGL 4.3 迁移到 DirectX 12。迁移的直接目的不是建设通用游戏引擎，而是消除 CBT 2024 复现中的图形接口差异，使 HLSL、资源屏障、无序访问资源、间接执行、GPU 时间戳和较新着色器能力能够按官方实现语义落地。

迁移采用以下原则：

1. 先冻结当前 OpenGL 版本及其基准数据，确保任何阶段都能回看原始结果。
2. 迁移期间使用短期双后端构建选项，逐项比较 OpenGL 与 DX12 行为；DX12 达到功能对等后停止继续扩展 OpenGL。
3. Classic CPU ROAM 和数据导向 CPU ROAM 的拓扑代码不重写，只替换网格上传与绘制路径。
4. 当前 GPU ROAM-like 和后续 CBT 2024 必须拥有自己的 GPU 常驻资源，不能继续依赖 DOD 拓扑快照作为最终架构。
5. 只抽象本项目需要的 GPU 能力，不设计覆盖所有图形接口的大型通用 RHI。
6. 在基础渲染、CPU 算法和基准测试功能对等之前，不开始修改 CBT 2024 的算法语义。

## 2. 当前耦合与迁移边界

| 模块 | 当前状态 | 迁移要求 |
|---|---|---|
| `platform/Window` | 同时拥有 SDL 窗口和 OpenGL 上下文 | 改为 SDL 窗口所有者；Windows 下暴露原生 `HWND`，交换链由 DX12 上下文管理 |
| `app/Application` | 直接加载 GLAD、清屏并调用 OpenGL 交换缓冲 | 改为驱动统一帧流程，具体清屏、提交和呈现交给 DX12 渲染上下文 |
| `gui/ImGuiLayer` | 使用 SDL2/OpenGL3 后端 | 切换到 SDL2/DX12 后端，并明确字体纹理描述符的所有权 |
| `render/Shader` | 封装 GLSL 编译、链接和 uniform | 替换为外部 HLSL、DXC 编译产物、根签名和管线状态对象 |
| `render/TerrainRenderer` | 直接管理 VAO、OpenGL 缓冲、纹理和间接绘制 | 改为管理 DX12 资源视图、根参数、管线状态、上传和绘制命令 |
| `ITerrainLodAlgorithm` | CPU 输入基本独立，但 GPU 输出暴露 OpenGL `uint32_t` 资源编号 | 保留算法输入与统计接口，重新设计 GPU 绘制数据包和资源生命周期约定 |
| `algorithms/gpu_roam` | 直接调用 OpenGL Compute Shader、屏障、查询和读回 | 迁移为 HLSL Compute Shader 和 DX12 命令记录；作为 CBT 前的计算管线验证用例 |
| CMake 与依赖 | 完整应用依赖 OpenGL、GLAD 和 ImGui OpenGL 后端 | 新增 DX12 后端选择、Windows SDK 链接、DXC 着色器编译和 ImGui DX12 后端 |
| 自动基准测试 | 已支持固定相机路径、CSV/Markdown 和 OpenGL Timer Query | 保留数据格式，增加后端、适配器、驱动和 DX12 时间戳字段 |

当前最重要的接口问题位于 [`ITerrainLodAlgorithm.h`](../../src/algorithms/ITerrainLodAlgorithm.h)：`TerrainLodRenderPacket` 直接暴露 OpenGL 资源编号。迁移后，CPU 算法仍输出 CPU 网格；GPU 算法输出 DX12 资源视图或后端绘制数据包，资源本身由算法实例跨帧持有。

## 3. 目标与非目标

### 3.1 迁移目标

- 使用 SDL2 创建窗口和处理输入，通过原生 `HWND` 创建 DXGI 交换链。
- 建立稳定的 DX12 设备、命令队列、命令列表、围栏和帧资源生命周期。
- 恢复高度图纹理、地形渲染、深度测试、线框、调试着色和 ImGui。
- 让 Classic CPU ROAM 和数据导向 CPU ROAM 在相同相机与参数下产生与迁移前一致的拓扑计数。
- 恢复自动烟雾测试、运行时基准测试、GPU 时间戳和 CSV/Markdown 输出。
- 提供 Compute Shader、无序访问缓冲、资源屏障、延迟读回和间接绘制能力。
- 为 CBT 2024 提供 Shader Model 6.6 能力检查、持久 GPU 内存池和完整资源验证环境。

### 3.2 非目标

- 不实现 Vulkan、Metal 或主机平台无关的完整 RHI。
- 不在迁移阶段重新设计 ROAM、DOD 或 CBT 算法。
- 不同时重写场景系统、相机、输入、资源格式或界面布局。
- 不把 OpenGL 与 DX12 的瞬时性能差异直接作为算法论文结论。
- 不在基础功能对等前引入预算调度器、闭包优先级或新的误差模型。

## 4. 目标架构

```text
Application
├── SDL 窗口、输入、相机与固定路径回放
├── ImGuiLayer（SDL2 + DX12）
├── D3D12Context
│   ├── 适配器、设备与功能检查
│   ├── 命令队列、交换链与帧资源
│   ├── 描述符堆、围栏与资源屏障
│   ├── 上传、读回与时间戳查询
│   └── DX12 调试层与设备移除诊断
├── TerrainRenderer
│   ├── 地形图形管线与根签名
│   ├── 高度图、纹理和固定渲染资源
│   ├── CPU 网格上传路径
│   └── GPU 常驻缓冲与间接绘制路径
└── ITerrainLodAlgorithm
    ├── Classic CPU ROAM -> CPU 网格
    ├── 数据导向 CPU ROAM -> CPU 网格
    ├── GPU ROAM-like -> GPU 绘制数据包
    └── CBT 2024 -> 持久 GPU 拓扑与 GPU 绘制数据包
```

### 4.1 资源所有权

- `D3D12Context` 拥有设备、交换链、命令队列、全局描述符堆和每帧同步对象。
- `TerrainRenderer` 拥有地形图形管线、纹理、深度缓冲和 CPU 网格上传资源。
- GPU LOD 算法拥有节点池、活动叶节点、候选列表、计数器、生成网格和间接命令缓冲。
- GPU 绘制数据包只借用资源视图，不转移资源所有权；其有效期至少持续到下一次算法 `Reset` 或资源重建。
- 读回资源按帧延迟消费，普通渲染帧不得同步等待 GPU，只允许验证模式显式阻塞。

## 5. 关键技术决策

### 5.1 平台与设备

- 第一目标平台为 Windows 10/11 x64。
- 保留 SDL2 窗口、事件、键盘和鼠标路径，不改为纯 Win32 应用。
- 使用高性能适配器优先策略，并提供适配器名称、显存和驱动信息记录。
- 基础渲染要求 D3D Feature Level 12_0；CBT 路径额外查询 Shader Model 6.6、64 位原子操作和所需无序访问能力。
- Debug 构建默认开启 DX12 调试层；GPU 验证作为可选开关，避免污染性能测试。

### 5.2 帧资源与同步

- 初始采用双缓冲帧资源，先保证资源生命周期清晰；功能稳定后再评估三缓冲。
- 每个帧资源独立持有命令分配器、常量上传区域和围栏值。
- 所有资源状态转换通过小型状态跟踪器记录，禁止在算法代码中散落未说明的屏障。
- 时间戳和统计读回至少延迟一帧，避免每帧 `WaitForFence`。

### 5.3 资源与描述符

- GPU 常驻网格、拓扑和纹理使用默认堆；上传和读回使用专用上传堆/读回堆。
- 分别管理 RTV、DSV 和可见的 CBV/SRV/UAV 描述符堆。
- 先采用明确的固定描述符布局，不在迁移阶段引入完整无绑定资源模型。
- 间接绘制使用 `ExecuteIndirect` 和固定命令签名，命令缓冲由 GPU 算法写入。

### 5.4 着色器与坐标约定

- 着色器从 C++ 内嵌字符串迁移为 `assets/shaders/dx12/*.hlsl` 文件。
- 使用 DXC 在构建阶段生成着色器字节码；Debug 保留调试信息，Release 使用优化配置。
- 统一规定矩阵主序、常量缓冲对齐和结构体打包方式，所有 CPU/HLSL 共享结构增加大小与偏移断言。
- DX12 深度范围为 `[0, 1]`，投影矩阵必须显式使用零到一深度约定，不能直接沿用 OpenGL 投影矩阵。
- 迁移测试必须覆盖纹理纵向、正反面绕序、法线方向、线框和深度精度差异。

### 5.5 短期后端选择

迁移期间新增构建变量：

```text
PARALLEL_ROAM_GRAPHICS_API=OpenGL|D3D12
```

它只用于选择窗口初始化、ImGui 后端、渲染器和 GPU 算法源文件，不要求运行时动态切换。DX12 达到功能对等并完成基准归档后，可删除 OpenGL 后端选择或仅保留在维护分支。

## 6. 公共接口迁移

### 6.1 保留部分

- `TerrainLodAlgorithmInfo`
- `TerrainLodAlgorithmCapabilities`
- `TerrainLodSettings`
- `TerrainLodBuildInput`
- CPU 网格数据和绝大多数 `TerrainLodStats`
- `BuildRenderData`、`Stats` 和 `Reset` 的算法生命周期

### 6.2 必须修改部分

`TerrainLodRenderPacket` 不再存放 OpenGL 资源编号。目标语义为：

```text
CpuMesh 模式：
    CPU 顶点与索引 -> TerrainRenderer 上传并绘制

GpuDirect 模式：
    DX12 顶点/索引资源视图 -> TerrainRenderer 直接绘制

GpuIndirect 模式：
    DX12 顶点/索引资源视图 + 间接命令资源 -> ExecuteIndirect
```

具体 C++ 类型在接口阶段确定，但必须满足：

- 不暴露 OpenGL `GLuint`；
- 不在每帧复制 GPU 资源所有权；
- 明确资源由算法还是渲染器持有；
- CPU 算法不需要包含 DX12 头文件；
- GPU 算法可以取得命令记录上下文和当前帧索引；
- benchmark 能区分 CPU 网格上传、GPU 计算、GPU 绘制和同步等待。

### 6.3 统计扩展

新增或预留以下字段：

- 图形后端与适配器名称；
- D3D12 功能级和 Shader Model；
- 图形、计算、上传和读回字节数；
- 各计算阶段时间戳；
- 资源屏障数量；
- 间接命令数量；
- GPU 常驻内存估算；
- 设备移除原因和验证错误计数；
- CBT 二分器容量、占用量、分配失败和传播轮数。

## 8. 测试矩阵

### 构建与运行

- Debug：开启 DX12 调试层和详细资源命名。
- RelWithDebInfo：用于开发期性能回归和 GPU 捕获。
- Release：用于最终论文和技术报告数据。
- GPU 烟雾测试：固定帧数、自动退出、失败返回非零状态码。
- 自动运行时基准：固定相机路径、预热、采样和结果落盘。

### 功能测试

- 窗口创建、缩放、最小化、恢复和 VSync 切换；
- 高度图与纹理加载；
- 相机、线框、光照和调试颜色；
- Classic、DOD、GPU ROAM-like 算法切换与重置；
- CPU 网格上传、GPU 直接绘制和 GPU 间接绘制；
- 高度图重载、算法重建和退出时资源释放；
- 设备功能不足时的明确错误信息。

### 对等性指标

- CPU 算法在固定输入下的节点数、三角形数、split/merge 数应完全一致；
- 规则地形和 CPU LOD 的顶点/索引数量应完全一致；
- 固定截图允许着色器舍入差异，但不得出现坐标、绕序、纹理或深度错误；
- DX12 Debug 层错误数必须为零；
- 连续运行不得出现持续显存增长；
- 性能测试不得包含未标注的每帧围栏等待或强制读回。

## 9. 主要风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 迁移与 CBT 同时推进 | 难以区分接口错误和算法错误 | 基础功能对等后才接入 CBT；CBT 原版路径先冻结语义 |
| OpenGL 与 DX12 坐标约定不同 | 地形翻转、深度错误、剔除错误 | 显式零到一投影、固定截图和绕序测试 |
| 资源状态或生命周期错误 | GPU 验证错误、随机崩溃、设备移除 | 统一状态跟踪、资源命名、围栏所有权和设备移除诊断 |
| 每帧同步等待 | 性能结果失真 | 帧延迟查询和读回；仅验证模式允许阻塞 |
| 描述符管理过早复杂化 | 延误基础功能 | 先使用固定布局和容量，CBT 需求明确后再扩展 |
| 双后端长期存在 | 维护成本翻倍 | 双后端只服务迁移验收，DX12 达标后归档 OpenGL |
| GPU 接口抽象过度 | 形成与研究无关的引擎工作 | 只封装缓冲、纹理、管线、命令、屏障、查询和间接绘制 |
| 基准口径变化 | 无法比较迁移前后结果 | 保留原字段、固定相机路径，并明确记录后端和同步成本 |

## 10. 交付物与完成定义

迁移阶段应产出：

- DX12 渲染与计算后端；
- 外部 HLSL 着色器和可复现的 DXC 构建流程；
- 更新后的 GPU 绘制数据包与资源生命周期说明；
- Classic、DOD、GPU ROAM-like 的 DX12 接入；
- DX12 烟雾测试、运行时基准和统计导出；
- OpenGL 与 DX12 功能对等验证记录；
- CBT 2024 所需功能检查与资源模型文档；
- 迁移前后截图、参数、日志和基准归档。

完成定义：DX12 主线能够在固定输入下稳定运行三套现有算法，CPU 算法拓扑计数与 OpenGL 基线一致，GPU 计算和间接绘制没有未说明的同步等待，Debug 层无错误，并具备接入 CBT 2024 完整计算流水线的所有基础能力。

## 11. 当前实施范围

阶段 0 至阶段 6 已完成：DX12 清屏与呈现、基础地形、ImGui、两套 CPU LOD、运行时基准和 GPU ROAM-like 均已接入。下一步进入阶段 7，独立实现 CBT 2024 的位域、归约树、二分器池、兼容链和完整 split/merge 流程；CBT 不复用 GPU ROAM-like 的拓扑逻辑，只复用已经验证的 DX12 计算、资源、屏障、计时和间接绘制基础设施。
